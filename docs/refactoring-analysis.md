# Анализ реализации и зоны для рефакторинга

## Контекст

Текущая реализация уже покрыта тестами и логически разделена на слои (`api` / `commands` / `report` / `tracker` / `pachca`), но есть несколько мест, где технический долг начнёт расти при увеличении количества команд, доменов и проектов.

Ниже — приоритизированные точки рефакторинга.

---

## 1) Безопасность вебхука: подпись сейчас не влияет на доступ (Высокий приоритет)

**Наблюдение:** в `app/api/webhook.py` при невалидной подписи пишется warning, но обработка команды и отправка сообщения продолжается.

**Риски:**
- endpoint можно дергать без валидной подписи;
- нагрузка на Tracker/Pachca и возможность злоупотребления ботом;
- размытая модель безопасности (лог есть, блокировки нет).

**Рефакторинг:**
- вынести проверку подписи в dependency/middleware;
- при невалидной подписи возвращать `401/403` и не переходить к бизнес-логике;
- добавить unit/integration-тесты на кейсы «валидная подпись / невалидная / отсутствует».

---

## 2) N+1 вызовы и отсутствие пакетной загрузки в агрегаторе (Высокий приоритет)

**Наблюдение:** `StatusAggregator` для каждого проекта запрашивает комментарии отдельно, а для обхода дерева портфелей делает последовательные вызовы.

**Риски:**
- деградация времени ответа при росте числа проектов;
- высокий шанс timeouts в webhook path;
- отсутствие контроля SLA по командам.

**Рефакторинг:**
- внедрить bounded concurrency (`asyncio.Semaphore`) для загрузки комментариев;
- рассмотреть метод клиента `list_comments_for_projects(project_ids)` (если API позволяет) или батч-стратегию;
- добавить кеш краткоживущего уровня (на 30-120 секунд) для списков портфелей в рамках процесса;
- ввести метрики: число запросов к Tracker на 1 команду, latency p50/p95.

---

## 3) Дублирование контрактов `TrackerClient` (Средний приоритет)

**Наблюдение:** протокол `TrackerClient` объявлен и в `app/tracker/client.py`, и в `app/report/aggregator.py`.

**Риски:**
- рассинхрон сигнатур при расширении API;
- неоднозначный source of truth для интерфейса.

**Рефакторинг:**
- оставить единый протокол в отдельном модуле (например, `app/tracker/protocols.py`);
- импортировать его в агрегатор и тестовые фейки;
- зафиксировать контракт в тесте совместимости (type-check или runtime-проверка минимального интерфейса).

---

## 4) Слишком много ответственности в `CommandRouter` (Средний приоритет)

**Наблюдение:** роутер одновременно парсит команды, управляет диалоговым состоянием, строит выборки и запускает рендер.

**Риски:**
- усложнение при добавлении новых команд/сценариев;
- высокий риск регрессии в state machine;
- сложно переиспользовать диалоговые сценарии.

**Рефакторинг:**
- выделить `CommandParser` (маппинг slash-команд -> intent);
- выделить `SelectionFlow` (FSM выбора домен/поддомен/команда);
- оставить в `CommandRouter` только orchestration;
- хранить описания команд декларативно (таблица/registry), чтобы убрать тройные циклы `for action` x `for level`.

---

## 5) In-memory состояние диалога без TTL/персистентности (Средний приоритет)

**Наблюдение:** `_pending` хранится в памяти процесса и очищается только при ручном завершении сценария или рестарте.

**Риски:**
- потеря контекста при деплое;
- зависшие диалоги без автоочистки;
- коллизии при горизонтальном масштабировании (несколько инстансов).

**Рефакторинг:**
- добавить abstraction для state storage;
- минимальный вариант: in-memory + TTL cleanup;
- production-вариант: Redis с TTL и ключом `(chat_id, bot_id)`.

---

## 6) Жестко зашитые значения в рендере (Низкий/Средний приоритет)

**Наблюдение:** текст про «старше 6 дней» зашит в `render_project_block`, хотя `freshness_days` конфигурируемый.

**Риски:**
- несоответствие текста реальной логике при изменении конфига;
- скрытые продуктовые ошибки.

**Рефакторинг:**
- передавать `freshness_days` в `ProjectSummary` или в параметры рендера;
- централизовать пользовательские строки в отдельный слой локализации/messages.

---

## 7) Обработка ошибок и наблюдаемость (Низкий/Средний приоритет)

**Наблюдение:** есть базовые логи, но нет явной таксономии ошибок и correlation между внешними вызовами.

**Риски:**
- сложно понимать, где bottleneck: Tracker, парсер статусов, Pachca API;
- трудно настраивать алерты по деградации.

**Рефакторинг:**
- добавить типизированные доменные исключения (`TrackerUnavailable`, `InvalidWebhookSignature`, `RenderError`);
- логировать отдельные этапы: `load_portfolios`, `load_projects`, `load_comments`, `render` с duration;
- добавить метрики (Prometheus/OpenTelemetry) и alert rules.

---

## 8) Потенциальные улучшения HTTP-клиентов (Низкий приоритет)

**Наблюдение:** клиенты корректные и компактные, но без retry/backoff/circuit-breaker.

**Риски:**
- временные 5xx/429 приводят к фейлу команды без повторной попытки;
- пиковые ошибки сразу видны пользователю.

**Рефакторинг:**
- добавить политику retry для idempotent-запросов;
- явно обрабатывать `429` с backoff;
- при необходимости — библиотека устойчивости (или собственный lightweight wrapper).

---

## Рекомендуемый порядок внедрения

1. Закрыть безопасность webhook (подпись -> hard fail).
2. Ускорить агрегатор (concurrency + метрики latency).
3. Разнести ответственность `CommandRouter` и унифицировать протокол клиента.
4. Вынести состояние диалогов в storage с TTL.
5. Привести рендер к конфигурируемым сообщениям и улучшить error handling.
