# План внедрения бота «Пачка ↔ Yandex Tracker»

**Документ:** план анализа требований и внедрения сервиса-бота, агрегирующего статусы проектов из Yandex Tracker и публикующего отчёты в мессенджере Пачка.

**Исходные документы:**
- `Архитектурное+ревью+сервис+для+_PMO_ (1).doc` — архитектурное ревью предыдущего сервиса (PMO helper, Jira + Confluence + Pachca). Используется как база архитектуры и стека.
- `Требования пачка:трекер бот (1).md` — бизнес-требования к новому боту (Yandex Tracker + Пачка).

---

## 1. Анализ архитектурного ревью (источник стека)

Предыдущий сервис PMO helper решал близкую задачу, но для Jira/Confluence. Из него переносим архитектурный каркас и технологические предпочтения.

### 1.1 Применимый стек

| Слой | Выбор из архивного сервиса | Комментарий для нового бота |
|---|---|---|
| Язык | Python 3 | Подтверждено: `DataProcessor`, `DataFrame`, парсеры комментариев — Python-экосистема |
| HTTP-сервис | REST API с вебхуком (`POST /webhook/pachea`) и служебными эндпоинтами | Переносим: webhook от Пачки + `/health` + внутренние триггеры |
| Очереди состояний | Redis (состояния пользователей, ожидание фильтра/периода) | Нужен для диалоговых сценариев, если появятся ветвящиеся команды |
| Планировщик | cron (регулярные отчёты раз в 2 недели) | В новых требованиях регулярных отчётов нет — только on-demand. Cron остаётся как задел на авторегламент |
| Интеграции | JiraClient, ConfluenceExporter, PachcaExporter | Заменяем Jira → Yandex Tracker, Confluence исключаем, Pachca-клиент переиспользуем |
| Деплой | Docker-контейнер, K8s pod, 128Mi–512Mi RAM, 0.1 CPU, 1–10 Gi disk | Для MVP укладываемся в минимальный pod |
| Сеть | HTTPS 443 вход/выход | Нужно согласовать исходящие на `api.tracker.yandex.net` и `api.pachca.com` |
| Аутентификация внешних API | Bearer/token в env (`JIRA_ACCESS_TOKEN`, `PACHCA_ACCESS_TOKEN`) | Для Tracker добавляем `TRACKER_OAUTH_TOKEN` + `TRACKER_ORG_ID` |
| Аутентификация входящих | `X-API-Key` заголовок для внутренних эндпоинтов | Сохраняем для защиты webhook-эндпоинта |
| Health check | `GET /health` с проверкой зависимостей | Переносим; проверяем Tracker и Пачку |
| Обработка ошибок | Retry Jira через 5 мин; Pachca/Confluence — не блокируют флоу | Переносим: Tracker retry, Пачка не валит процесс |
| Логирование | Структурные логи + предупреждения на пропусках задач | Переносим; добавляем correlation id вебхука |

### 1.2 Применимые архитектурные паттерны

- **Слоистая модель:** `Client → DataProcessor → ReportGenerator → Exporter`. Для нового бота: `TrackerClient → StatusParser → ReportBuilder → PachcaExporter`.
- **Разделение use case'ов:** регулярные (cron) и ad-hoc (webhook). В MVP ad-hoc достаточно; cron-часть оставляем расширяемой.
- **State machine для команд:** в старом сервисе пользователь мог уходить в состояния `waiting_filter`/`waiting_period`. Новым командам (п. 6 требований) стейт-машина пока не нужна — все команды одношаговые. Закладываем возможность позже добавить выбор «какого именно» домена/поддомена/команды.
- **Graceful degradation:** сбой публикации не должен валить сборку данных.
- **Документирование через UC** (UC-001, UC-002…): переиспользуем формат для новых кейсов.

### 1.3 Что НЕ переносим

- Confluence-интеграция (`ConfluenceExporter`, `/report_conf`, `/report_for_a_period_confluence`).
- Jira-специфика: фильтры `40996/40997`, поля `customfield_13200`, JQL, формат `#ReportDelivery`.
- Регулярные 2-недельные отчёты — в текущих требованиях не заявлены.
- Периодные отчёты (`/report_for_a_period_*`) — не заявлены.

## 2. Анализ бизнес-требований (новый бот)

### 2.1 Суть

Бот по команде в Пачке формирует агрегированный статус-отчёт по проектам в Yandex Tracker. Агрегация идёт по иерархии портфелей: **Домен → Поддомен → Команда (= портфель нижнего уровня) → Проекты**.

### 2.2 Ключевые бизнес-правила (кратко)

1. Иерархия источника — только портфели Трекера; «команды» как сущность не используются, команда = портфель нижнего уровня.
2. Один проект может быть в нескольких портфелях, при этом на уровне домена/поддомена выводится **один раз**, на уровне команды — во всех привязанных командах.
3. Статус проекта берётся из:
   - поля «Статус» проекта,
   - **последнего** комментария с тегом `#WeeklyStatus` в разделе «Обновления», не старше 6 дней.
4. Шаблон комментария: `#WeeklyStatus`, `Comments:`, `DL по решению:`.
5. Цветовой маппинг статусов: По плану 🟢, Есть риски 🟡, Заблокирован 🔴, не актуально/отсутствует ⚪.
6. Если последний `#WeeklyStatus` старше 6 дней — статус помечается как «неизвестен/отсутствует», сам текст показывается как справочный.
7. Форматирование комментария сохраняется (переносы, списки, базовая разметка).
8. Команды (всего 10): 3 × report + 3 × list + 3 × risk + `/help`.

### 2.3 Состав отчёта

| Поле | Источник в Трекере |
|---|---|
| Название + ссылка | `summary` / `self` URL проекта |
| Ответственный | Поле «Ответственный» (`lead` или кастомное) |
| Статус | Поле «Статус» проекта |
| Комментарий | Последний `#WeeklyStatus` с сохранением форматирования |
| DL по решению | Из тела комментария, парсинг строки «DL по решению:» |

### 2.4 Отличия от старого сервиса

| Аспект | PMO helper (старый) | Новый бот |
|---|---|---|
| Источник данных | Jira | Yandex Tracker |
| Модель группировки | Команды через `customfield_13200` | Иерархия портфелей Трекера |
| Публикация | Confluence + Pachca | Только Пачка |
| Триггеры | cron + команды | Только команды |
| Тег комментария | `#ReportDelivery`, `#WeeklyStatus` | Только `#WeeklyStatus` |
| Окно актуальности | 10 дней | 6 дней |
| Команды | ~8 (включая периоды и Confluence) | 10 (domain/subdomain/team × report/list/risk + help) |

## 3. Что нужно получить / настроить для разработки

### 3.1 На стороне Yandex Tracker

**Доступ и идентификация**
- [ ] OAuth-токен с правами на чтение проектов, портфелей и комментариев (`TRACKER_OAUTH_TOKEN`).
- [ ] `X-Org-ID` (или `X-Cloud-Org-ID`) — идентификатор организации.
- [ ] Сервисный аккаунт с доступом ко всем нужным портфелям (правила видимости в Трекере).

**Данные и API, которые понадобятся**
- [ ] API портфелей: получение дерева «Домен → Поддомен → Команда» (`GET /v2/entities/portfolio/_search`, `GET /v2/entities/portfolio/{id}`).
- [ ] API проектов: список проектов внутри портфеля (`GET /v2/entities/project/_search` c фильтром по `parentEntity` / `portfolio`).
- [ ] API комментариев: `GET /v2/issues/{issueKey}/comments` или эквивалент для сущности «проект»/«обновления».
- [ ] Поля проекта: `summary`, `status`, `lead` (ответственный), ссылки.
- [ ] Формат хранения «Обновлений» (updates) проекта — уточнить, это отдельная сущность или обычные комментарии.

**Согласование с владельцем Трекера**
- [ ] Эталонный портфель-пример для e2e-тестов.
- [ ] Список допустимых значений поля «Статус» проекта и их соответствие трём бизнес-статусам.
- [ ] Правила именования портфелей, по которым бот будет отличать «домен / поддомен / команду» (если это не определяется структурой дерева).
- [ ] Ограничения по rate limit API Трекера → стратегия кэширования.

### 3.2 На стороне Пачки

**Бот и доступ**
- [ ] Создать бота/интеграцию в пространстве; получить `PACHCA_ACCESS_TOKEN`.
- [ ] Зарегистрировать webhook URL сервиса и секрет верификации (`X-API-Key` или подпись).
- [ ] Добавить бота в целевые чаты (домен/поддомен/команда) и согласовать политику приватности.

**Команды и UX**
- [ ] Согласовать набор slash-команд (п. 6 требований).
- [ ] Проверить поддержку форматирования сообщений (markdown, переносы, списки) — требование п. 8.
- [ ] Договориться о лимитах длины сообщения и стратегии разбивки длинных отчётов.
- [ ] Подтвердить, как бот узнаёт, какой **именно** домен/поддомен/команда запрошен: по ID в команде (`/show_team_report <id>`), по выбору из списка после `_list`, либо по привязке чата к портфелю.

### 3.3 Инфраструктура

- [ ] Docker-образ + K8s-манифест (128Mi–512Mi RAM, 0.1 CPU).
- [ ] Публичный HTTPS-эндпоинт для webhook (reverse proxy / ingress).
- [ ] Хранилище: для MVP достаточно кэша в памяти; Redis вводим при появлении стейт-машины или для кэша API Трекера.
- [ ] Секреты: хранилище переменных окружения (`TRACKER_OAUTH_TOKEN`, `TRACKER_ORG_ID`, `PACHCA_ACCESS_TOKEN`, `WEBHOOK_API_KEY`).
- [ ] Мониторинг: `/health`, логи, метрики количества отчётов / ошибок.

```
                │  Пачка чат     │
 user /cmd ───► │  webhook POST  │
                └──────┬─────────┘
                       │ HTTPS + X-API-Key
                ┌──────▼─────────────────────┐
                │  API (FastAPI / aiohttp)   │
                │  ├─ /webhook/pachca        │
                │  ├─ /health                │
                │  └─ CommandRouter          │
                └──────┬─────────────────────┘
                       │
          ┌────────────┼──────────────────────┐
          ▼            ▼                      ▼
  ┌─────────────┐ ┌───────────────┐ ┌──────────────────┐
  │ TrackerClient│ │ StatusParser │ │ ReportBuilder    │
  │ (REST/OAuth) │ │ #WeeklyStatus│ │ aggregate by lvl │
  └──────┬───────┘ └──────┬───────┘ └────────┬─────────┘
         │                │                  │
         ▼                ▼                  ▼
   Yandex Tracker   in-memory cache   PachcaExporter ──► Пачка
```

**Модули (Python-пакеты):**

| Модуль | Ответственность |
|---|---|
| `app.api` | FastAPI-приложение, роутинг webhook, аутентификация |
| `app.commands` | Распознавание команд, маршрутизация в сценарии |
| `app.tracker` | Клиент Yandex Tracker, работа с портфелями/проектами/комментариями |
| `app.status` | Парсер `#WeeklyStatus`, определение актуальности (≤6 дней), нормализация статуса |
| `app.report` | Построение агрегированных отчётов по уровням (domain/subdomain/team), дедупликация мульти-портфельных проектов |
| `app.pachca` | Клиент Пачки: отправка сообщений с сохранением форматирования и пагинацией |
| `app.config` | Настройки, секреты, маппинг статусов |
| `app.health` | Health-check зависимостей |
| `app.logging` | Структурные логи + correlation id |

## 6. Пошаговый план внедрения

### Этап 0. Подготовка (≈ 2–3 дня)
1. Получить доступы: токен Трекера, токен Пачки, `Org-ID`, доступ к эталонному домену/поддомену/команде.
2. Уточнить в Трекере формат «обновлений» проекта (комментарии vs отдельная сущность).
3. Согласовать список допустимых значений поля «Статус» и их маппинг на 3 бизнес-статуса.
4. Согласовать способ выбора домена/поддомена/команды в команде Пачки (аргумент, выбор из списка, привязка к чату).
5. Развернуть репозиторий: структура пакетов, `pyproject.toml`, линтер, pre-commit, CI.

### Этап 1. Каркас сервиса (≈ 2 дня)
1. FastAPI-приложение: `/webhook/pachca`, `/health`, middleware `X-API-Key`.
2. Конфиг через `pydantic-settings`, логирование через `structlog`.
3. Docker-образ + манифест деплоя.
4. Health-check заглушка + smoke-тесты.

### Этап 2. Tracker-клиент (≈ 3–4 дня)
1. Модуль `app.tracker`: OAuth-клиент на `httpx.AsyncClient`.
2. Методы: `get_portfolio_tree()`, `list_projects_in_portfolio(id)`, `get_project(id)`, `get_project_updates(id)`.
3. Определение уровня портфеля (домен/поддомен/команда) по глубине вложенности.
4. Дедупликация мульти-портфельных проектов для уровней домен/поддомен.
5. Юнит-тесты на фикстурах ответов API.
6. Базовый кэш ответов (TTL 1–5 мин) для снижения нагрузки на Трекер.

### Этап 3. Парсер статусов (≈ 1–2 дня)
1. Модуль `app.status`: парсинг тела комментария (`#WeeklyStatus`, `Comments`, `DL по решению`).
2. Правило «не старше 6 дней».
3. Маппинг статусов и эмодзи.
4. Сохранение форматирования (перенос строк, списки) при переносе в Пачку.
5. Юнит-тесты на разных шаблонах комментариев.

### Этап 4. Построение отчётов (≈ 2–3 дня)
1. `ReportBuilder.build_list(level)` — UC-101.
2. `ReportBuilder.build_full(level, id)` — UC-102.
3. `ReportBuilder.build_risk(level, id)` — UC-103.
4. Правила отображения мульти-портфельных проектов (на командном уровне — во всех, на уровне домена/поддомена — один раз).
5. Юнит-тесты с фикстурами «дерево портфелей + проекты + комментарии».

### Этап 5. Pachca exporter и командный слой (≈ 2 дня)
1. `app.pachca`: отправка сообщений с форматированием и пагинацией при превышении лимита длины.
2. `app.commands`: роутер 10 команд + `/help`.
3. Обработка неизвестных команд (сообщение + подсказка `/help`).
4. Интеграционный тест end-to-end на тестовом чате.

### Этап 6. Надёжность и деплой (≈ 2 дня)
1. Retry-политика для Трекера (экспоненциальный backoff, 2 попытки).
2. Graceful degradation: ошибка Пачки не валит обработку команды, ошибка Трекера возвращает понятное сообщение пользователю.
3. Метрики: кол-во команд, время ответа, ошибки по типам.
4. Полный health-check (Tracker + Pachca).
5. Раскатка в stage, приёмочное тестирование на эталонном портфеле, прод-раскатка.

### Этап 7. Документация и передача (≈ 1 день)
1. README: запуск, переменные окружения, примеры команд.
2. Use-case документ в формате старого ревью (UC-101…UC-106).
3. Runbook: типовые инциденты и действия.

**Итого по MVP:** ≈ 13–17 рабочих дней одним разработчиком.

## 7. Открытые вопросы (требуют согласования до старта)

1. **Обновления проекта в Трекере** — это обычные комментарии или отдельная сущность updates? От ответа зависит API-метод и парсер.
2. **Как бот определяет целевой портфель в команде?** Варианты: аргумент `/show_domain_report <id>`, двухшаговый сценарий (сначала `_list`, потом выбор), привязка чата к порталу. Текущие требования допускают любой — нужно зафиксировать.
3. **Маппинг статусов Трекера → 3 бизнес-статуса** (По плану / Есть риски / Заблокирован) — нужно явное соответствие.
4. **Ответственный** — это системное поле `lead` или кастомное поле проекта.
5. **Права доступа**: должен ли бот учитывать права пользователя, вызвавшего команду, или отдавать всё, что видно сервисному аккаунту.
6. **Лимиты сообщений Пачки** — максимальная длина и стратегия разбивки (несколько сообщений / вложение / тред).
7. **Регламент** — остаются ли регулярные авто-рассылки? В требованиях их нет, но из архитектурного ревью видно, что это типичный сценарий.
8. **Хранение истории отчётов** — нужно ли сохранять snapshot'ы для аудита (в старом сервисе это было на стороне Confluence).

## 8. Риски и митигация

| Риск | Влияние | Митигация |
|---|---|---|
| API Трекера не отдаёт иерархию портфелей в один запрос | Долгие ответы, rate limit | Кэш на 1–5 мин; батчинг; прогрев по cron |
| Формат `#WeeklyStatus` нарушается пользователями | Пустые отчёты | Толерантный парсер + сообщение об ошибке формата в отчёте |
| Большой чат-отчёт не помещается в одно сообщение Пачки | Потеря части данных | Пагинация / отправка в тред / аттач |
| Смена схемы полей Трекера | Падение парсера | Контрактные тесты на фикстурах; версии API в конфиге |
| Недоступность Пачки | Команда падает | Ретраи + информативный ответ; не блокирует логирование |
| Утечка токенов | Компрометация данных | Хранение в секретах, маскирование в логах, ротация |

## 9. Критерии приёмки MVP

- [ ] Все 10 команд обрабатываются и возвращают ответ в пределах SLA (п. 5).
- [ ] Отчёт по эталонному домену совпадает с ручной выборкой из Трекера.
- [ ] Мульти-портфельный проект на уровне команды выводится во всех командах, на уровне домена/поддомена — один раз.
- [ ] Проекты без `#WeeklyStatus` за 6 дней помечаются ⚪ с текстом последнего комментария как справкой.
- [ ] Форматирование комментария (переносы, списки) сохраняется в Пачке.
- [ ] `/health` показывает состояние Tracker и Пачки.
- [ ] Логи содержат correlation id по webhook-запросу.
- [ ] Секреты не попадают в репозиторий и в логи.
