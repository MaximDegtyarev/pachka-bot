# Пользовательская инструкция: pachka-bot

Бот строит отчёты по портфелям из Yandex Tracker и присылает их в чат Пачки. Работает на иерархии **Домен → Поддомен → Команда → Проект**.

---

## 1. Как работает бот (коротко)

1. Вы пишете в чате команду, например `/show_team_report`.
2. Если у вас несколько команд/поддоменов/доменов, бот присылает нумерованный список и ждёт от вас **цифру** — номер нужного портфеля.
3. Бот собирает проекты выбранного портфеля, читает последние комментарии с тегом `#WeeklyStatus` в каждом проекте и формирует итоговый отчёт.
4. Отчёт приходит в тот же чат в виде Markdown-сообщения.

Статус проекта в отчёте складывается из двух источников:

- **Бизнес-статус** в самом Tracker (поле проекта «Статус»): «По плану», «Есть риски», «Заблокирован».
- **Свежий комментарий** с тегом `#WeeklyStatus`. Если такого комментария нет или он старше **6 дней**, бот показывает статус как «⚪ Статус неизвестен», даже если в Tracker статус выставлен — потому что без свежего комментария нельзя верить, что статус актуальный.

---

## 2. Список команд

### 2.1. Общие

| Команда | Что делает |
|---|---|
| `/help` | Показать список всех команд. |

### 2.2. Списки портфелей

| Команда | Что делает |
|---|---|
| `/show_domain_list` | Все домены, настроенные у бота (со ссылками на Tracker). |
| `/show_subdomain_list` | Все поддомены внутри настроенных доменов. |
| `/show_team_list` | Все команды внутри настроенных доменов. |

Списки нужны, чтобы быстро увидеть «что вообще есть» и скопировать название нужного портфеля.

### 2.3. Полные отчёты (все проекты портфеля)

| Команда | Уровень |
|---|---|
| `/show_domain_report` | По домену. |
| `/show_subdomain_report` | По поддомену. |
| `/show_team_report` | По команде. |

### 2.4. Только проекты с рисками (🟡)

| Команда | Уровень |
|---|---|
| `/show_domain_risk` | По домену. |
| `/show_subdomain_risk` | По поддомену. |
| `/show_team_risk` | По команде. |

В эти отчёты попадают **только** проекты со статусом «Есть риски». Заблокированные проекты сюда **не** включены — для них отдельные команды.

### 2.5. Только заблокированные проекты (🔴)

| Команда | Уровень |
|---|---|
| `/show_domain_blocked` | По домену. |
| `/show_subdomain_blocked` | По поддомену. |
| `/show_team_blocked` | По команде. |

### 2.6. Только проекты по плану (🟢)

| Команда | Уровень |
|---|---|
| `/show_domain_on_track` | По домену. |
| `/show_subdomain_on_track` | По поддомену. |
| `/show_team_on_track` | По команде. |

### 2.7. Кросс-командные проекты

| Команда | Уровень |
|---|---|
| `/show_cross_domain` | По домену. |
| `/show_cross_subdomain` | По поддомену. |
| `/show_cross_team` | По команде. |

Показывает проекты, у которых в Tracker выставлен тег **`cross`**. Чтобы проект попал в этот отчёт, достаточно добавить тег `cross` на карточку проекта в Tracker.

## 3. Диалог выбора портфеля

Команды уровня `_report`/`_risk`/`_blocked`/`_on_track` запрашивают портфель пошагово — всегда начиная с домена:

| Команда | Шаги |
|---|---|
| `/show_domain_*` | выбрать домен |
| `/show_subdomain_*` | выбрать домен → выбрать поддомен |
| `/show_team_*` | выбрать домен → выбрать команду |

Пример диалога с несколькими доменами:

```
Вы:   /show_team_report
Бот:  Выберите домен:

      1. B2B PMO
      2. B2C PMO

      Введите номер:
Вы:   1
Бот:  Выберите команду:

      1. Team Alpha
      2. Team Beta
      3. Team Gamma

      Введите номер:
Вы:   2
Бот:  **Отчёт команды: Team Beta**
      ...
```

Правила:

- Пока диалог ждёт число, **число** → выбор; **любая slash-команда** (`/help`, `/show_...`) → отмена диалога и выполнение новой команды.
- Если ввести число вне диапазона, бот подскажет правильный интервал.
- Любой шаг, где доступен ровно **один** вариант, пропускается автоматически. Например, если у бота настроен только один домен — шаг выбора домена в `/show_team_*` и `/show_subdomain_*` не показывается.
- Если в выбранном домене нет команд/поддоменов — бот сообщит об этом и завершит диалог.
- Диалоги у разных пользователей/чатов независимы.

## 4. Как выглядит отчёт

Пример строки проекта:

```
**[Chatbot для PMO](https://tracker.yandex.ru/pages/projects/123)**
- Ответственный: Иван Иванов
- Статус: 🟢 По плану
- Дедлайн: 2026-12-31
- Комментарий:
  - выкатили MVP в staging
  - ждём обратную связь от пилотной команды
- DL по решению: 25.04.2026
```

Что здесь что:

- **Название проекта** — кликабельная ссылка на проект в Tracker.
- **Ответственный** — руководитель проекта (lead) из Tracker. `—` если не назначен.
- **Заказчик** — поле «Клиенты» проекта в Tracker. Показывается через запятую, если заполнено; строка не отображается, если список пуст.
- **Статус** — бизнес-статус с эмодзи (см. §5).
- **Дедлайн** — поле `end` проекта в Tracker (если заполнено).
- **Комментарий** — текст из последнего `#WeeklyStatus` (поле `Comments:`). Отбивается в маркированный список с отступом, многострочный текст сохраняет форматирование.
- **DL по решению** — поле из `#WeeklyStatus` (дедлайн принятия решения / milestone).

Если статус устарел или не заполнен — в конце блока будет соответствующая пометка (см. §6).

---

## 5. Цвета статусов

| Эмодзи | Бизнес-статус | Когда показывается |
|---|---|---|
| 🟢 | По плану | В Tracker статус «По плану» И есть свежий `#WeeklyStatus` (≤6 дней). |
| 🟡 | Есть риски | В Tracker статус «Есть риски» И есть свежий `#WeeklyStatus`. |
| 🔴 | Заблокирован | В Tracker статус «Заблокирован» И есть свежий `#WeeklyStatus`. |
| ⚪ | Статус неизвестен | Статус в Tracker один из трёх распознаваемых, но `#WeeklyStatus`-комментарий отсутствует или устарел. |
| — | Проект скрыт | Статус в Tracker не является ни «По плану», ни «Есть риски», ни «Заблокирован» (например, «Черновик», «В работе», «Запущен»). Такой проект **не появляется в отчётах совсем**. |

Важно: **даже если в Tracker выставлен корректный статус, но комментарий устарел/отсутствует, бот покажет ⚪**. Смысл: руководитель PMO видит в отчёте только то, что владелец проекта недавно подтвердил.

---

## 6. «Статус не заполнен» vs «Данные устарели»

В конце блока проекта может появиться одна из двух пометок:

- **_Статус не заполнен (добавьте комментарий с #WeeklyStatus)._**
  Так бывает для только что созданных проектов — в них ещё вообще нет комментариев с тегом `#WeeklyStatus`.

- **_Данные устарели (старше 6 дней)._**
  Есть старый комментарий; его текст показан как справка, но верить ему нельзя, пока не напишут новый.

---

## 7. Как правильно заполнить `#WeeklyStatus`

Владельцу проекта нужно добавить в **комментарии к проекту** в Tracker следующий блок (раз в неделю или чаще):

```
#WeeklyStatus
Comments: Коротко о прогрессе за неделю.
Могу писать на нескольких строках.
DL по решению: 25.04.2026
```

Правила:

- Тег `#WeeklyStatus` обязателен (регистр не важен).
- Поле `Comments:` — произвольный текст, может быть многострочным, все последующие строки до следующего «Comments:»/«DL по решению:» считаются частью комментария.
- Поле `DL по решению:` — опциональное, подставится в отчёт отдельной строкой.
- Бот берёт **самый свежий** комментарий с этим тегом. Старые можно не удалять.

### 7.1. Что изменить в самом проекте

Бот смотрит на бизнес-статус проекта (поле «Статус») в Tracker. Чтобы отчёт показал 🟢/🟡/🔴, в Tracker проект должен быть в одном из состояний:

| Статус в UI Tracker | Эмодзи в отчёте | Виден в отчёте |
|---|---|---|
| По плану | 🟢 | да |
| Есть риски | 🟡 | да |
| Заблокирован | 🔴 | да |
| Черновик / В работе / Запущен / любое другое | — | **нет, скрыт** |

Дедлайн в отчёте — это поле **«Дата окончания» (end)** на карточке проекта в Tracker.

## 8. Типичные сценарии

### 8.1. Руководитель: «Что у моей команды в красной зоне?»

```
/show_team_blocked
→ выбрать домен (если доменов несколько)
→ выбрать команду
```

### 8.2. Руководитель поддомена: «По каким проектам есть риски?»

```
/show_subdomain_risk
→ выбрать домен (если доменов несколько)
→ выбрать поддомен
```

### 8.3. PMO: «Какой общий портрет домена?»

```
/show_domain_report
→ выбрать домен (или сразу отчёт, если домен один)
```

### 8.4. Проверить свежесть статусов

Используйте `/show_domain_report` и обратите внимание на проекты с пометкой _«Статус не заполнен»_ или _«Данные устарели»_ — по ним нужно попросить владельцев обновить комментарий.

### 8.5. Посмотреть, какие вообще есть поддомены/команды

```
/show_subdomain_list
/show_team_list
```

## 9. Ограничения и тонкости

- **Скрытые проекты.** Проекты со статусом «Черновик», «В работе», «Запущен» и любым другим, кроме трёх отслеживаемых, полностью исключаются из всех отчётов. Это поведение намеренное: такие проекты не подлежат еженедельному статусингу.
- **Дедупликация.** Если один и тот же проект привязан к нескольким командам, то:
  - в отчётах по **команде** он появится в каждой из них;
  - в отчётах по **поддомену/домену** — ровно один раз.
- **Freshness = 6 дней.** Это настраивается администратором бота (переменная `STATUS_FRESHNESS_DAYS`), но по умолчанию — 6.
- **Markdown.** Бот присылает форматированный Markdown. В Пачке он рендерится как заголовки, списки, ссылки.
- **Длинные сообщения.** Если отчёт очень большой (>4000 символов), бот сам разобьёт его на несколько сообщений по пустым строкам.
- **Состояние диалога.** После рестарта сервиса незавершённые «ожидания цифры» сбрасываются — просто повторите команду.

---

## 10. Если что-то не работает

- Бот не отвечает на `/help` → вероятно, в настройках чат-бота в Пачке заполнено поле «Команды», из-за чего Пачка не шлёт вебхуки на остальные команды. Нужно **очистить поле «Команды»** в настройках бота.
- Бот отвечает на собственные сообщения (зацикливается) → в настройках бота в Пачке включите «Игнорировать свои сообщения».
- В отчёте все проекты ⚪ → проверьте, что (а) комментарии с `#WeeklyStatus` есть, (б) они моложе 6 дней, (в) в проекте выставлен один из трёх распознаваемых статусов.
- Дедлайн не отображается → поле «Дата окончания» (`end`) пустое на карточке проекта.
- В `/show_domain_list` нет вашего домена → попросите администратора добавить ID домена в `PORTFOLIO_DOMAIN_IDS`.
