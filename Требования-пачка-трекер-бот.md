# Создаем файл business-requirements-bot.md
content = """
# Бизнес-требования к боту отчетности (Пачка + Yandex Tracker)

## 1. Цель

Бот предназначен для автоматического сбора и агрегации статусов проектов из Yandex Tracker и формирования отчетов по различным уровням управления:

- портфель домен;
- портфель поддомен;
- портфель команда.

## 2. Данные

### 2.1 Источник структуры

Источник структуры — иерархия портфелей в Yandex Tracker:

```

Домен (портфель)
├── Поддомен (портфель)
│   └── Портфель команды
│       └── Проекты

```

### 2.2 Правила

- Команды как отдельная сущность не используются
- Портфель нижнего уровня = команда
- Каждый проект обязан находиться внутри портфеля команды
- Один проект:
  - может принадлежать нескольким портфелям
  - автоматически входит в поддомен и домен

## 3. Источник статусов

### 3.1 Где оставляется статус

Статус оставляется:

- **в проекте**
- **В поле "Статус"**
- **В комментарии** в разделе **"Обновления"**

### 3.2 Формат комментария

**Обязательное требование:**

Комментарий должен содержать тег: `#WeeklyStatus`

**Если тег отсутствует:**
- комментарий игнорируется ботом

### 3.3 Выбор статуса

Для каждого проекта бот:

1. находит все комментарии с тегом `#WeeklyStatus`
2. выбирает **последний комментарий** (если он оставлен не позднее 6 дней)

**Шаблон статуса проекта:**

```

\#WeeklyStatus
Comments:
DL по решению:

```

### 3.4 Статусы по которым собираем информацию

- По плану
- Есть риски  
- Заблокирован

### 3.5 Правила заполнения

| Поле | Описание | Обязательность |
|------|----------|----------------|
| Comments | Текстовое описание текущего состояния | Обязательно |
| DL по решению | Дата/срок принятия решения по риску/блокеру | Обязательно для risk/blocked |

## 4. Логика сбора данных

### 4.1 Привязка данных

```

Комментарий → Проект → Портфель команды → Портфель Поддомен → Портфель Домен

```

### 4.2 Алгоритм

1. **Пользователь выбирает уровень отчета:**
   - домен / поддомен / команда
   
2. **Бот:**
   - находит соответствующий портфель
   - получает список всех вложенных проектов
   
3. **По каждому проекту:**
   - получает комментарии
   - фильтрует по `#WeeklyStatus`
   - берет последний
   
4. **Формирует агрегированный отчет** и направляет в чат

## 5. Требования к отчету

### 5.1 Полный отчет

Для каждого проекта в отчете отображается:

| Поле | Источник |
|------|----------|
| **Название проекта** | Название + [гиперссылка на проект] |
| **Ответственный** | Поле "Ответственный" |
| **Статус** | Поле "Статус" |
| **Комментарий** | Последний `#WeeklyStatus` |
| **DL по решению** | Из комментария |

## 6. Пользовательские команды

| Команда | Описание | Результат |
|---------|----------|-----------|
| `/show_domain_report` | Отчет по домену | Отчет по всем проектам домена |
| `/show_subdomain_report` | Отчет по поддомену | Отчет по проектам поддомена |
| `/show_team_report` | Отчет по команде | Отчет по проектам портфеля команды |
| `/show_domain_list` | Список доменов | Список портфелей верхнего уровня |
| `/show_subdomain_list` | Список поддоменов | Список вложенных портфелей |
| `/show_team_list` | Список команд | Список портфелей команд |
| `/show_domain_risk` | Риски домена | Проекты домена с статусом `at risk` |
| `/show_subdomain_risk` | Риски поддомена | Проекты поддомена с статусом `at risk` |
| `/show_team_risk` | Риски команды | Проекты команды с статусом `at risk` |
| `/help` | Помощь | Список доступных команд |

## 7. Логика работы со статусами

### 7.1 Маппинг статусов

| Статус | Цвет |
|--------|------|
| По плану | 🟢 Зеленый |
| Есть риски | 🟡 Желтый |
| Заблокирован | 🔴 Красный |
| Не обновлен | ⚪ Серый |

### 7.2 Правило неактуальности

**Если данные старше 6 дней:**

- Статус: **"Статус неизвестен или отсутствует"**
- Дополнительно: показывается текст последнего комментария как справочная информация

## 8. Требования к отображению комментариев

Комментарий должен отображаться **с сохранением форматирования**:

- ✅ Переносы строк
- ✅ Списки  
- ✅ Базовая разметка

## 9. Агрегация данных

**Используются все проекты внутри выбранного портфеля.**

### 9.1 Мультипортфельные проекты

**Один проект может находиться в нескольких портфелях.**

### 9.2 Правила отображения

| Уровень отчета | Правило отображения |
|----------------|---------------------|
| **Команда** | Проект отображается во **всех** привязанных командах |
| **Поддомен/Домен** | Проект отображается **только один раз** |

## 10. Общие ограничения

- ✅ Бот работает только по принципу **агрегации данных** из Yandex Tracker
- ✅ Комментарии **без тега `#WeeklyStatus`** игнорируются
- ✅ Если актуальный комментарий отсутствует → проект помечается как **неактуальный**
- ✅ Отчет должен быть **понятным** для управленческого просмотра

**Дата создания:** 17.04.2026
"""
